# Open Brain — Design Guidelines

**Date:** 2026-04-17
**Status:** Light guidelines, not a full design system
**References:** [CONSTRAINTS-2026-04-17.md](./CONSTRAINTS-2026-04-17.md), [TECH-STACK-2026-04-17.md](./TECH-STACK-2026-04-17.md)

---

## 1. Vibe

**"A functional tool that happens to live in 1987."**

The aesthetic is lightly 80s / cyberpunk — phosphor glows, monospace status readouts, a scan-line moment here and there — but **function dominates**. The user is reading and writing their own notes, often for hours. The theme is a *texture*, not a costume. Think IBM terminal, not Blade Runner billboard.

Rule of thumb: if the theme is ever fighting legibility, the theme loses.

---

## 2. Theme Modes

### Switcher
- Three modes: **System** (default), **Light**, **Dark**.
- Setting persisted in local config + `.openbrain/config.json` so it syncs across devices.
- Implemented via `[data-theme="light"|"dark"]` on `<html>`; when set to System, no attribute is set and CSS uses `prefers-color-scheme`.
- Switch lives in a settings menu, not as a prominent toggle — you set it once.

### Philosophy
- **Dark mode is the canonical design target.** That's where the 80s feel lives.
- **Light mode is a muted, paper-like translation** of the same system — never a simple color inversion. It's for sunny cafes and for users who just prefer it.

## 3. Color

Exact hex values TBD during build; these are the **roles**. We'll define them as Tailwind 4 `@theme` tokens so both modes share semantic names.

### Dark mode
| Role | Character |
|---|---|
| `--bg` | Near-black, not pure black. Slight warm blue cast (think CRT shadow, not OLED void). |
| `--bg-raised` | ~6% lighter — for cards, toasts, modals. |
| `--fg` | Soft phosphor white/off-white. Not pure `#fff`. |
| `--fg-muted` | Lower-contrast text for metadata, timestamps. |
| `--accent` | **One** phosphor accent (cyan or magenta — pick during first visual pass, do not use both). Used for focus rings, active tab underline, chat cursor, link hover. |
| `--accent-glow` | Same hue, semi-transparent, used for subtle outer glows on focus. |
| `--border` | 1px hairline, low-contrast. |
| `--danger` | Muted red — never neon. Errors must feel calm. |
| `--warn` | Muted amber. |
| `--ok` | Muted green. |

### Light mode
| Role | Character |
|---|---|
| `--bg` | Warm off-white, paper-like (`#f7f4ec`-ish). Not pure white. |
| `--bg-raised` | Slightly cooler/lighter card surface. |
| `--fg` | Near-black, warm. |
| `--fg-muted` | Desaturated warm gray. |
| `--accent` | Desaturated sibling of the dark-mode accent. If dark uses cyan, light uses a dusty teal, not bright cyan. |
| `--accent-glow` | Not used in light mode — glows look wrong on light backgrounds. |
| `--border` | 1px warm gray hairline. |
| `--danger / --warn / --ok` | Muted; same semantic hue family as dark. |

### Rules
- **One accent only.** Resist the urge for pink + cyan + purple neon. Pick one and commit. A second color can appear later for charts.
- **Never pure `#000` or `#fff`.** Both are harsh and break the paper/phosphor feel.
- **WCAG AA minimum** for all text. AAA for body text. This is a tool people will stare at.

## 4. Typography

- **Body / UI:** a neutral sans-serif (default pick: **Inter** via system `font-sans`, falling back to the OS UI font). Unstyled, unopinionated.
- **Monospace:** used for timestamps, status readouts, model names, paths, code, and the chat input cursor. Default pick: **JetBrains Mono** or the system `ui-monospace` stack. This is where the 80s character lives.
- **Markdown editor (CodeMirror):** monospace. Always.
- **No display / decorative fonts.** If a single heading ever needs to look "cyberpunk," it does so via an accent color or a subtle rule above it, not a typeface.

Sizing: use a modular scale (1.125× or 1.2×). Keep the type ramp short — 5 sizes max.

---

## 5. Iconography

- **Line icons only.** Default pick: **Lucide** (already Svelte-friendly, tree-shakeable).
- **Monospace status glyphs** for terminal-style readouts: `▲ ▼ ◆ ● ○ ◇ →`. Used in status bars and command palettes, not in primary UI buttons.
- No 3D / skeuomorphic / colored-gradient icons. The UI stays flat.

---

## 6. Layout

- **Three-tab persistent layout** (Chat / Browse / Memory) — nav bar on top at desktop, bottom on mobile.
- **Thin status bar** at the very bottom (desktop) or just above the tab bar (mobile). Monospace. Shows:
  - Sync status: `▲ synced 2s ago` / `◆ 3 changes pending` / `○ offline`
  - Model status: `gemma-4b ready` / `gemma-4b loading 43%`
  - Extraction queue: `3 notes pending index`
  - Click any of these → relevant detail panel.
- **1px hairline borders**, not drop shadows, for separation. Shadows show up rarely — only on modals and toasts — and they're tight, not soft.
- **No rounded corners above 4px.** Cards, inputs, buttons all have the same modest radius. Pills and badges can be fully rounded.
- **Density over whitespace** — but not cramped. A second brain is an information tool; aerating it into a marketing page wastes screen.

---

## 7. Motion

- **Short, linear, unspringy.** Default transition: 150ms ease-out. No bounces.
- **Honor `prefers-reduced-motion`.** Replace transitions with instant state changes.
- **Streaming tokens are the animation** in Chat. Don't layer additional motion on top.
- **The only "cyberpunk" motion moments:**
  - Blinking cursor on the chat input (1s cycle, square wave — feels like a terminal).
  - Brief phosphor glow pulse on focused inputs (subtle; ~200ms; fades, doesn't loop).
  - Optional scan-line overlay during model download progress screens. Off by default in Light mode. Togglable off entirely in settings.

---

## 8. Loading States

### Principle
**Invisible where possible; useful feedback when not.** Never show a spinner "just in case."

### Tiers
1. **Under ~150ms** — no indicator. The eye won't register it.
2. **150ms – 1s** — optimistic UI: assume success, render the result, reconcile silently on error. Works for: saves, note creation, sidecar updates, sync commits.
3. **1s – 10s** — inline skeleton of the expected shape. No centered spinners; no full-screen overlays. A loading chat response is just the streaming tokens appearing.
4. **> 10s** — explicit progress UI with a readable status line. Used for: model download, initial repo clone, first-run indexing. Progress % + what it's doing right now (`"Downloading gemma-4-4b (2.1 GB / 4.3 GB)"`).

### What we don't do
- Full-screen spinners.
- Global loading bars at the top of the page (the status bar handles it).
- Shimmer skeletons on every element. Skeletons are reserved for visually load-bearing content (opening a note, loading a chat history).

## 9. Error Handling

### Principle
**Retry invisibly when sensible; minimal toast otherwise; full details to console.**

### Categories
| Error | UX |
|---|---|
| Transient network (5xx, timeout) | Silent retry with exponential backoff. Status bar shows `◇ syncing…` during retries. Surface to user only if all retries fail. |
| GitHub rate limit | Silent wait-and-retry. Status bar: `◇ rate-limited, resuming in 34s`. |
| Conflict on sync | Not an error — handled by the three-tier conflict resolution. Tier 2 surfaces a resolver UI, not a toast. |
| Auth expired | Inline banner in the affected tab: "Sign in again to sync." Not a toast. |
| Model load failure | In-place error in the Chat tab with a Retry button. Not a toast. |
| Unexpected / unknown | Toast (see below) + full stack to console. |

### Toast format
- **Position:** bottom-center desktop, top of content area mobile.
- **Size:** single line, plus an optional "Details" action that opens devtools hint (`console.error` has the full context).
- **Content:** human-readable verb + noun. `"Couldn't save note. Changes kept locally."` — never `"Error: unhandled promise rejection: TypeError: …"`.
- **Dismissal:** auto-dismiss after 6s unless an action is present; tap to dismiss.
- **Throttling:** identical errors within 30s collapse to a single toast with a count (`"Sync failed (×3)"`).

### Console
- Every surfaced error is also `console.error`'d with a structured object:
  ```ts
  console.error("[openbrain] sync.push failed", { attempt, path, err });
  ```
- A hidden debug panel (triggered by `?debug=1` or a key chord) shows the recent error log for users who don't know devtools exists.

## 10. Accessibility

- **Keyboard:** every action reachable. Tab order matches visual order. `Cmd/Ctrl+K` opens a command palette (terminal feel + a11y win).
- **Focus:** always visible. The phosphor-glow focus ring is the accessibility signal and the aesthetic moment at the same time.
- **Screen reader:** semantic HTML; tabs use ARIA tablist; live region announces toasts and streaming status.
- **Contrast:** AAA body text, AA everywhere else.
- **Reduced motion:** honored; scan-line overlay and cursor blink both disabled.
- **Zoom:** layout holds up to 200% text zoom.

## 11. The 80s Moments — A Short Checklist

Places where the theme shows up, in order of prominence:

1. Status bar — monospace, glyph-driven, always visible. **This is the #1 vibe carrier.**
2. Focus glow on inputs.
3. Blinking terminal-style cursor in the Chat input.
4. Monospace timestamps and paths throughout.
5. Command palette (Cmd/Ctrl+K) with a monospace prompt.
6. Subtle scan-line overlay on model-download and initial-clone screens only.
7. Optional — rare — a `[READY]` / `[SYNCING]` chip somewhere in the Memory tab.

**Places we intentionally avoid the theme:**
- Body copy (legibility).
- Markdown content (it's the user's writing, not ours).
- Error states (errors should feel calm, not dramatic).
- Mobile layout edge cases (screen size is too precious for flourish).
